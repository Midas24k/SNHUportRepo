# SNHUportRepo

Project Summary

For my CS 210 portfolio artifact, I selected my Project Three submission, the Corner Grocer item frequency program. This project was designed to solve the problem of analyzing grocery purchase records from an input file and giving the user meaningful information through a menu-driven program. The program allows the user to search for the frequency of a specific grocery item, display a full list of all grocery items with their frequencies, and generate a text-based histogram to visually represent item frequency.

This project demonstrates my ability to use C++ to build a functional program that reads data from a file, processes that data, and presents the results in a clear way for the user. It also reflects my understanding of control structures, functions, file handling, and readable program design.

What I Did Well

One thing I did particularly well was organizing the program around a clear user menu. The menu makes the program easy to use because the user can choose exactly what type of information they want to see. I also did well with processing the grocery data and turning it into useful output, especially the histogram. The histogram makes the data easier to understand because it gives the user a quick visual representation instead of only listing numbers.

I also focused on making the output readable and professional. The program gives clear prompts, handles the user’s menu choices, and formats the frequency information in a way that makes sense.

Areas for Enhancement

One area where I could enhance my code is modularization. In future improvements, I would separate the program into multiple files, such as a header file, implementation file, and main driver file. This would make the code easier to maintain, test, and reuse in future projects. A cleaner file structure would also make the project look more professional in a portfolio.

I could also improve input validation. For example, I could make the program better at handling invalid menu choices, blank item searches, or unexpected data in the input file. These improvements would make the program more secure and reliable because it would be less likely to crash or return incorrect results when the user enters unexpected input.

Another enhancement would be improving efficiency by using a map or unordered map to store item names and frequencies. This would make searching and frequency counting faster and more organized, especially if the input file became much larger.

Challenges and Support Network

The most challenging part of this project was making sure the program correctly counted item frequencies and displayed the information in multiple ways. It required me to think carefully about how the data was being read, stored, and reused throughout the program. I overcame this by breaking the problem into smaller pieces, testing each menu option separately, and checking that the output matched the expected results.

The tools and resources I added to my support network include zyBooks, course resources, instructor feedback, C++ documentation, and debugging tools in my development environment. I also learned the value of testing small sections of code before trying to fix the whole program at once. That helped me find issues faster instead of chasing bugs all over the place like a detective in a grocery aisle.

Transferable Skills

Several skills from this project will transfer to future courses and real-world programming work. File input and output are important in many applications because programs often need to read, store, and process data. Menu-driven design is also useful because it teaches how to create programs that respond to user choices.

This project also strengthened my understanding of loops, conditionals, functions, data structures, and debugging. These skills will be useful in future programming courses, software development projects, and larger applications where clean logic and reliable data handling are important.

Maintainability, Readability, and Adaptability

I made the program maintainable and readable by using clear function names, consistent formatting, and comments to explain important sections of the code. I also tried to keep the program organized so that each part had a clear purpose.

The program is adaptable because the same structure could be expanded for other types of inventory tracking or data analysis. For example, instead of grocery items, the program could be modified to track warehouse products, attendance records, sales data, or other repeated values from a file. With better modularization and input validation, this project could become a stronger reusable tool in the future.
